## Backend.

What we’re testing: system thinking, API design, data modelling, and whether your code holds up when it is reviewed for audit, not for aesthetics.

## Q1. Tamper-Evident Append-Only Log Service

Build a service where every event is written once and can never be edited or deleted. The chain must be self-verifiable so a third party can prove the log has not been tampered with. Node and Express with PostgreSQL preferred. SQLite acceptable if you justify it in the README.

REQUIRED

- POST /log accepts { actor, action, payload }. Each entry stores a SHA-256 chain hash that links to the previous entry. No edits, no deletes.
- GET /log/:id returns the entry plus a chain-verification status.
- GET /verify scans the entire chain and returns pass or fail, with the first broken entry id if any.
- GET /export returns a filtered JSON export (by date range and actor).
- Structured logging using pino or winston. A rate limit on POST. Database migrations checked in.
- A small auth layer (API key in a header is fine).

## STRETCH (OPTIONAL)

- Merkle-tree style batching so /verify stays fast on $true$ entries.
- A CLI command (npm run verify) that runs the full chain check.
- A docker-compose so we can boot the whole thing with one command.

## Q2. Profile-to-Recommendation API

A user sends a structured profile and you return the top three items from a catalogue that match them, with reasoning. Pick the domain yourself. Build a non-trivial catalogue of at least 15 items. We are watching how you model eligibility rules and how clearly you explain a match.

## REQUIRED

- POST/recommend takes a profile (you decide the fields, minimum five attributes) and returns a ranked top three with a one-paragraph reason for each match.
- GET/items (admin token protected) returns full CRUD.
- GET /explain/:item_id returns the eligibility logic in plain language.
- At least five unit tests covering boundary, missing-field, and no-match cases.
- A seed script that loads the $true$ item catalogue into the database.

## STRETCH (OPTIONAL)

- A /subscribe webhook so a user is notified when newly added items match their profile.
- A caching layer with a sensible TTL on the recommend endpoint.
- OpenAPI spec auto-generated from the routes.

→ Pick ONE, Q1 or Q2, not both. We will not grade the second one.
